# Requirements Archive: OpenClaw Dashboard v1.4.0

**Archived:** 2026-03-16
**Status:** ✅ Shipped
**Milestone Goal:** Make the Agents view much easier to scan so the operator can see who is working, blocked, or idle at a glance, understand what each working agent is actually doing, and see what idle agents could credibly pick up next.

## Outcomes

### Agent Roster Clarity

- [x] **AGENT-01**: Operator can see every agent's name, status, current task, duration, and room/team in the default Agents view
  Outcome: shipped in Phase 8 through roster-first working, blocked, and idle cards.
- [x] **AGENT-02**: Operator can understand who is working, blocked, or idle within a few seconds of opening the Agents view
  Outcome: shipped in Phase 8 with explicit triage sections and scan-first presentation.
- [x] **AGENT-03**: Agents view prioritizes roster legibility over secondary intelligence/detail clutter
  Outcome: shipped in Phases 8 and 10 by simplifying the default layout and preserving deeper rails as secondary context.

### Status-Based Triage

- [x] **TRIAGE-01**: Agents are grouped into separate **Working**, **Blocked**, and **Idle** sections
  Outcome: shipped in Phase 8.
- [x] **TRIAGE-02**: Blocked agents appear in a clearly distinct section that makes intervention needs obvious
  Outcome: shipped in Phase 8 with stronger blocked emphasis and inline context.
- [x] **TRIAGE-03**: Idle agents appear in a clearly distinct section that makes assignment opportunities obvious
  Outcome: shipped in Phase 8 with a dedicated idle roster and compact queue.

### Idle Suggestions

- [x] **IDLE-01**: Idle agents can show a lightweight suggested next task hint in the Agents view
  Outcome: shipped in Phase 8, then broadened in Phase 10 with advisory repo/research sources.
- [x] **IDLE-02**: Suggested tasks follow a clear ranking rule: same room first, then role-fit, then any ready unowned task
  Outcome: shipped in Phase 8 through the initial advisory ranking model and preserved after provenance work landed.
- [x] **IDLE-03**: When no reasonable next task exists, the Agents view shows a clear "no obvious assignment" state instead of a fake suggestion
  Outcome: shipped in Phase 8 with explicit fallback copy.

### Idle Assignment Queue

- [x] **QUEUE-01**: Operator can see a compact queue of idle agents and their suggested next task candidates
  Outcome: shipped in Phase 8.
- [x] **QUEUE-02**: Idle-assignment queue stays compact and scannable instead of duplicating full Mission Control task management
  Outcome: shipped in Phase 8 and preserved in Phase 10's concise coordination redesign.
- [x] **QUEUE-03**: Idle-assignment queue is clearly suggestive, not an automatic assignment engine
  Outcome: shipped in Phase 8 and reinforced in Phase 10 advisory wording.

### Blocked Agent Emphasis

- [x] **BLOCK-01**: Blocked agents show enough reason/context in the Agents view for the operator to understand why they are blocked
  Outcome: shipped in Phase 8.
- [x] **BLOCK-02**: Blocked agents show blocked duration or similarly useful time context
  Outcome: shipped in Phase 8 through inline time and state presentation.
- [x] **BLOCK-03**: Blocked state emphasis helps the operator identify intervention needs without opening Mission Control first
  Outcome: shipped in Phase 8 via the dedicated blocked section and emphasis styling.

### Responsive Layout

- [x] **RESP-01**: Desktop Agents view uses a dense, operator-friendly layout optimized for fast comparison across many agents
  Outcome: shipped in Phase 8.
- [x] **RESP-02**: Smaller screens adapt into a more card-like or stacked presentation without losing core status clarity
  Outcome: shipped in Phase 8 and validated in browser checks.
- [x] **RESP-03**: Responsive layout preserves the same mental model across screen sizes: Working, Blocked, Idle, then suggestions
  Outcome: shipped in Phase 8.

### Legacy Repo Task Retirement

- [x] **ARCH-01**: Repo-task flows no longer depend on `task-center`, `agent-launchpad`, or `agent-workflow` as live systems of record
  Outcome: shipped in Phase 9 through the archive-era local runtime and mutation contract.
- [x] **ARCH-02**: Repo-bound task data from those systems is archived, migrated, or removed cleanly so operators are not looking at stale duplicate repo tasks
  Outcome: shipped in Phase 9 through bundled-data cleanup and archive-aware sample state.
- [x] **ARCH-03**: Any remaining runtime/task integrations in the dashboard reflect the cleaned post-archive architecture rather than the older three-repo split
  Outcome: shipped in Phase 9 and reinforced by updated Mission Control copy plus the runbook.

### Task Cleanup

- [x] **CLEAN-01**: Repo-related tasks are removed from the surviving task-center dataset
  Outcome: shipped in Phase 9.
- [x] **CLEAN-02**: Personal research tasks that should remain in task-center are preserved as the valid `TQ-XXX` set
  Outcome: shipped in Phase 9 through the retained `TQ-091` and `TQ-101` items.
- [x] **CLEAN-03**: The cleanup path makes it obvious which tasks were kept, archived, or removed
  Outcome: shipped in Phase 9 through mission notes, docs, and the updated verification flow.

### Live Work Provenance

- [x] **PROV-01**: Working agents show the repo or personal-research thread they are actively serving instead of only a generic room or model label
  Outcome: shipped in Phase 10 through explicit `workloads` and provenance cards.
- [x] **PROV-02**: Provenance comes from live OpenClaw/Codex/Discord session metadata, especially `#repo-work` coding threads and `#intake` personal research threads
  Outcome: shipped in Phase 10 with exact repo/intake cases in the snapshot and bundled demo fixtures.
- [x] **PROV-03**: When an agent has multiple active sessions, the Agents view exposes enough session/task context to show actual workload instead of collapsing everything into one ambiguous desk state
  Outcome: shipped in Phase 10 with multi-session workload entries and UI rendering.

### Advisory Suggestion Sources

- [x] **SUG-01**: Idle suggestion queue looks at any tracked GSD repo with open roadmap or pipeline work that can be started or unblocked
  Outcome: shipped in Phase 10 through repo-plan advisory sourcing.
- [x] **SUG-02**: Idle suggestion queue also considers actionable personal research tasks from intake-style threads
  Outcome: shipped in Phase 10 through surviving research-task advisory candidates.
- [x] **SUG-03**: Suggestions remain clearly advisory and explain why a repo/research item is being suggested
  Outcome: shipped in Phase 10 with source labels, ranking reasons, and non-owning language.

### Concise Coordination View

- [x] **SIMP-01**: Content below the Working / Blocked / Idle sections is redesigned into a concise coordination surface with clear purpose
  Outcome: shipped in Phase 10 with the `Coordination brief` default.
- [x] **SIMP-02**: Secondary analytics, rails, or detail panels are reduced, merged, or deferred so the default view does not overwhelm the operator
  Outcome: shipped in Phase 10 by shortening the lower-half default and moving deeper detail lower in the scan path.
- [x] **SIMP-03**: The default Agents page emphasizes only the most helpful next-action information about agents, blockers, and likely handoffs
  Outcome: shipped in Phase 10 through `Active workloads` and `Advisory next moves`.
