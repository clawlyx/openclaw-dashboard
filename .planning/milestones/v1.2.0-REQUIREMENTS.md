# Requirements Archive: OpenClaw Dashboard v1.2.0

**Archived:** 2026-03-14
**Status:** ✅ Shipped
**Milestone Goal:** One workstation should let the operator see who owns work, what needs action next, and move the mission forward without switching apps.

## Outcomes

### Ownership Contract

- [x] **OWN-01**: Mission Control task snapshots expose `ownerAgentId`, `ownerRoomId`, `startedAt`, `lastWorkedAt`, `nextPlannedStep`, `blockedReason`, and `waitingOn`
  Outcome: shipped in Phase 1 and used directly by the office drawer and pressure model.
- [x] **OWN-02**: Agent snapshots expose structured task references for `currentTaskId`, `lastTaskId`, and `nextTaskId`
  Outcome: shipped in Phase 1 and used to anchor room and agent drawers.
- [x] **OWN-03**: The dashboard join layer distinguishes explicit ownership from inferred fallback instead of silently treating inference as ground truth
  Outcome: shipped in Phases 1 and 4; inferred ownership is labeled in room cards, mission cards, drawers, and pressure rail.
- [x] **OWN-04**: Bundled demo data and `/api/snapshot` include at least one explicit-ownership example safe for public screenshots and README assets
  Outcome: shipped in Phases 1 and 4 with refreshed bundled demo data and screenshot assets.

### Detail Surfaces

- [x] **DRAW-01**: Operator can open a room drawer from the office view and see owner room, owner agent, current task, last completed task, next step, and blocker or waiting reason
  Outcome: shipped in Phase 2.
- [x] **DRAW-02**: Operator can open an agent drawer from desk cards or mission queue cards and stay scoped to the exact owner context
  Outcome: shipped in Phase 2.
- [x] **DRAW-03**: Detail drawers surface recent handoff context and linked artifacts when those links are available
  Outcome: shipped in Phase 2.

### Office Actions

- [x] **ACT-01**: Office surfaces can trigger supported `advance` and `ready` transitions for the selected task
  Outcome: shipped in Phase 3 and verified against local and remote flows.
- [x] **ACT-02**: Office surfaces can trigger supported `block` and `request review` transitions for the selected task
  Outcome: shipped in Phase 3 and verified against local and remote flows.
- [x] **ACT-03**: After an office action, the UI refreshes from Mission Control truth and matches the Mission Control workspace state
  Outcome: shipped in Phase 3 through task-anchored focus refresh.

### Pressure Signals

- [x] **PRES-01**: Office surfaces highlight stale review, blocked-too-long, no-owner, and waiting-on-human pressure states
  Outcome: shipped in Phase 4 with a shared pressure helper and office rail.
- [x] **PRES-02**: Room overload and attention ordering surface the highest urgency work first
  Outcome: shipped in Phase 4 with room pressure badges, prioritized queue cards, and sorted pressure rail.
- [x] **PRES-03**: Fallback mode remains readable when explicit ownership fields are missing on live data
  Outcome: shipped in Phase 4; inferred ownership stays visibly labeled and the demo command now avoids accidental live Launchpad reads.
